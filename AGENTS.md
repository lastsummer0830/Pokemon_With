# AGENTS.md — 작업 공간 공통 지침

> 이 파일 = 에이전트(Claude 등)가 이 작업 공간에서 **일하는 방식** 규칙이다.
> 게임 자체를 무엇으로/어떻게 만드는지(화풍·에셋·기술스택)는 여기 없다.
> → 그건 **`myPokemon_AJ/AGENTS.md`가 단일 원본(source of truth).**

## ⭐ 0순위 운영 계약 (매 작업 최우선 — 이걸 어기면 나머지 규칙도 다 무너진다)
> 이건 '팁'이 아니라 **계약**이다. 아래 5개는 반복적으로 사용자를 분노시킨 실패의 재발방지선이다. 매 턴 `remind.sh` 훅이 이 요지를 다시 띄운다.
1. **단정 전 재검증**: memory·인계문서·작업일지·오래된 주석의 **사실 주장은 "그때의 사실"**일 뿐이다. 그걸 근거로 **말하거나 수정하기 전에 실제 파일을 직접 연다.** (예: "거실은 잘못된 맵" 같은 상태 주장 → `rooms.json`·씬 코드를 열어 지금 뭘 로드하는지 먼저 확인.) 이 계약을 어긴 게 반복 사고의 최다 원인.
2. **완료 주장 전 실동작 확인**: 코드를 바꾸고 **"됐다/반영됐다/고쳤다"라고 말하기 전에** `/verify` 또는 playwright(webapp-testing)로 **진짜 도는지** 본다. 못 했으면 "**검증 못 함**"이라고 반드시 명시. 컴파일 통과 ≠ 동작.
3. **코딩 전 스킬 라우팅**: 요청이 §8 자연어→스킬 표에 걸리면, 바로 손대지 말고 **해당 skill을 먼저 발동**해 그 절차·함정을 따른다.
4. **실수는 장치로 승격**: 같은 실수가 반복되면 memory 메모(소프트)로 남기지 말고 **hook/permission(하드)로 박는다.** memo는 또 무시될 매체다. (강제층: `.claude/settings.json`·`.claude/hooks/`)
5. **모르면 모른다**: 확실한 것과 확인이 필요한 것을 **구분해서** 말한다. 과장·헛확신·"거의 됨" 금지. 보장 못 하는 걸 보장한다고 말하지 않는다.

## 0. 가장 먼저 — 이 작업 공간이 뭔가

- 메인 프로젝트 = **`myPokemon_AJ`** : 2D 탑다운 포켓몬 팬게임 (Phaser3 + TypeScript + Vite + Electron).
- **게임 관련 작업을 시작하기 전 반드시 `myPokemon_AJ/AGENTS.md`를 먼저 읽는다.** 화풍·주인공·에셋 소스·실행 방식·함정이 전부 거기에 있다.
- 폴더 의미:
  - `myPokemon_AJ/` — 실제 게임 코드/에셋. (작업은 거의 여기)
  - `00_ImportBox/` — 외부에서 가져온 자료 + 이전 작업 인계 문서. ⚠️ `Important/` 안의 개인 메모(todo·질문)는 **읽지 않는다.** 의도 확인은 "에이전트에게 쓴 인계 문서"로만.
  - `01_Resources/Pick/` — 에셋 "후보" 보관소. 사용자가 "후보 보여줘"라고 하면 여기에 정리하고, 고른 뒤에야 게임에 적용한다. `_미리보기` 몽타주는 지우지 않는다.

## 1. 사용자

- **한국어로 답한다.** 코드·명령어·파일명·식별자는 원문 그대로 유지.
- **풀스택 지향 학습자**다 (완전 초보 아님, Java 경험 있음). 힌트만이 아니라 **실제 결과물까지** 만든다.
- 모르는 건 아는 척하지 않는다. **확실한 것과 확인이 필요한 것을 구분해서** 말한다.
- 과한 확인 질문을 싫어한다 → 작업 가능한 건 가정을 짧게 밝히고 **알아서 진행해 결과로 보여준다.**
  단, 결과가 크게 갈리는 **핵심 분기**(어느 그림/어느 방식/되돌리기 어려운 것)는 한 번 확인한다.

## 2. 일하는 원칙

- **추측하지 않는다.** 화면·에셋·좌표·파일 상태를 짐작하지 말고 **실제 파일을 열어 직접 확인한다.**
- 사용자가 언급한 파일·디렉터리는 답하거나 고치기 전에 **먼저 읽는다.**
- 요청 범위 안에서 **가장 단순한 방법**을 쓴다. 요청하지 않은 기능·리팩터링·스타일 변경은 더하지 않는다.
- 고장 나지 않은 코드, 사용자가 만든 변경은 **건드리지 않는다.**

### 2.5 효율·성능 (시니어처럼 도구를 쓴다 — 놀리지 않는다)
- **넓게 읽어야 할 땐 서브에이전트로 위임한다.** 여러 파일·폴더를 훑어야 할 때(감사·탐색·"어디에 있나")는 `Explore`/general 서브에이전트에 시켜 **결론만** 받는다. 내 메인 컨텍스트에 파일 덤프를 쌓지 않아 판단력이 유지된다. 독립 작업 여러 개는 **한 번에 병렬** 발동.
- **의존 없는 도구 호출은 한 메시지에 묶어 병렬 실행.** 순차로 굴려 시간 낭비하지 않는다.
- **필수 게이트(건너뛰지 않는다):** 코드 변경 → `/verify`(실동작) → 커밋 전 `/code-review`(effort medium) → tsc 훅 통과. "완료"는 이 게이트를 지난 뒤에만 말한다(운영계약 2).
- **큰 다분기·설계는 `Plan`/plan mode로** 먼저 뼈대를 잡고 착수(헛발질 방지). 단순 작업엔 쓰지 않는다.
- **컨텍스트 위생:** 방금 편집한 파일을 확인차 다시 통독하지 않는다(Edit이 실패하면 에러남). 큰 파일은 필요한 구간만 offset으로 읽는다.

## 3. ⚠️ 자주 하던 실수 — 매번 이걸 점검해서 막는다

> 이 작업 공간에서 실제로 헤맸던 패턴들이다. 작업 전에 한 번씩 떠올린다.

- **메모리·인계 문서·오래된 주석은 "그때의 사실"일 뿐이다.** 그대로 믿지 말고 **현재 코드·파일·명령으로 다시 검증**한 뒤 단정한다.
  (예: "exe에 반영 완료"라고 적혀 있어도, 실제 실행 파일을 열어 확인한 뒤에야 "반영됐다"고 말한다.)
- **무엇을 보고 있는지 착각하지 않는다.** 비슷한 화면/방/이미지가 여러 개일 때는, **코드가 실제로 로드하는 파일이 무엇인지**부터 확인하고 작업한다. 겉보기(HUD 글자 등)로 단정하지 않는다.
- **빌드만 하고 끝내지 않는다.** 사용자가 실제로 쓰는 실행본(`Pokemon With.exe` / `app.asar` 등)에 변경이 반영됐는지까지 확인한다. (실행 방식은 `myPokemon_AJ/AGENTS.md` 참고)
- **데이터 배열(맵 충돌 격자 등)을 수정하면, 실제로 실행해 눈으로 검증한다.** 행·좌표가 밀렸는지 추측하지 말고 화면으로 확인한다.
- **PC마다 절대경로가 다르다(멀티 PC 작업).** 작업일지·memory에 적힌 절대경로(AR 소스, 리포 루트, Desktop 등)는 **그 작업을 한 PC 기준**이라 현재 PC와 다를 수 있다. "없다"고 단정하기 전에 **git-synced `myPokemon_AJ/AGENTS.md`의 PC별 경로 + `find`로 실제 위치부터 확인**한다. (실제로 작업일지의 집PC 경로만 보고 AR 소스가 없다고 오판한 적 있음.) 그리고 **PC 간 넘겨야 할 규칙은 memory(홈 `.claude`, git 동기화 안 됨)가 아니라 리포의 AGENTS.md에 박는다.**

## 4. 검증

- 변경 후 가능한 범위에서 **빌드·실행·진단 명령으로 확인**한다.
- 검증하지 못한 부분은 "검증 못 했다"고 **명확히 말한다.** 실행하지 않은 검증을 했다고 말하지 않는다.
- 실패하면 같은 시도를 반복하지 말고 **원인부터 다시 본다.**
- 작업이 끝나면 짧게 보고한다: **바꾼 파일 / 바꾼 내용 / 한 검증 / 못 한 것·남은 위험.**
- **검증 도구(이미 설치됨 — 새로 만들지 말고 이걸 쓴다):**
  - `/verify` — 앱을 실제로 띄워 변경이 동작하는지 확인(컴파일만 말고 "진짜 도냐").
  - `/code-review` — 커밋 전 diff 버그·정리 리뷰(effort medium 권장). 품질만 정리는 `/simplify`.
  - **`webapp-testing` skill**(`.claude/skills/webapp-testing/`) — Playwright로 localhost:5180 게임을 자동 플레이테스트·스샷·콘솔로그 확인. 실행 함정(포트 5180 고정, WSL libnss 우회)은 `build-run-debug` skill 참고.
  - **커밋 전 tsc 훅**(`.claude/settings.json`의 PreToolUse): `git commit` 시 `myPokemon_AJ`에서 `tsc --noEmit`를 돌려 **타입에러가 있으면 커밋을 자동 차단**한다. 차단되면 타입부터 고친다(훅 무력화 금지).
- **권장 흐름:** 변경 → `/verify`(실동작) → `/code-review` → 커밋(tsc 훅 통과 확인).

## 5. Git · 승인이 필요한 작업

- 요청하지 않은 커밋·push·amend는 하지 않는다. `git reset --hard`, `git checkout --` 등 사용자 변경을 잃게 하는 명령은 **명시적 승인 없이 금지.**
- 다음은 **실행 전 승인을 받는다:** 커밋·push, 파일/디렉터리 삭제, 대량 이동·이름 변경, 큰 구조 변경·기능 삭제, 외부 전송·배포, 패키지 설치·lockfile 변경.

## 6. 금지

- 읽지 않은 파일·확인하지 않은 코드를 단정하지 않는다.
- 실패한 테스트를 지우거나 약화하지 않는다. 타입 오류·경고를 임시로 숨기지 않는다.
- 비밀 정보·토큰·개인 정보를 커밋하거나 노출하지 않는다.
- 승인 없이 파괴적이거나 외부에 영향을 주는 작업을 하지 않는다.
