> 에이전트가 이 작업 공간에서 **일하는 방식** 규칙. 게임 자체(화풍·에셋·기술스택)는 **`myPokemon_AJ/AGENTS.md`**가 정본. 규칙 계층·강제층은 `CLAUDE.md` 참조.

## ⭐ 0순위 운영 계약 (매 작업 최우선 — 반복적으로 사용자를 분노시킨 실패의 재발방지선)
1. **단정 전 재검증**: memory·인계문서·작업일지·오래된 주석의 사실 주장은 "그때의 사실"일 뿐. **말하거나 고치기 전에 실제 파일을 직접 연다.** (예: "거실은 잘못된 맵" → `rooms.json`·씬 코드를 열어 지금 뭘 로드하는지 먼저 확인.) 이 계약 위반이 반복 사고 최다 원인.
2. **완료 주장 전 실동작 확인**: "됐다/반영됐다/고쳤다" 말하기 전 `/verify` 또는 webapp-testing(playwright)로 진짜 도는지 본다. 못 했으면 "**검증 못 함**" 명시. 컴파일 통과 ≠ 동작.
3. **코딩 전 스킬 라우팅**: 요청이 게임 `AGENTS.md §8` 자연어→스킬 표에 걸리면 손대기 전 **해당 skill 먼저 발동**.
4. **실수는 장치로 승격**: 같은 실수 반복되면 memo(또 무시될 매체) 말고 **hook/permission(하드)로 박는다**(`.claude/settings.json`·`hooks/`).
5. **모르면 모른다**: 확실한 것과 확인 필요한 것을 구분. 과장·헛확신·"거의 됨" 금지. 보장 못 하는 걸 보장한다 말하지 않는다.
6. **규칙은 스코프에 맞게 — 전역오염 금지**: 프로젝트별 규칙(훅·스킬·경로)은 그 프로젝트 `.claude/`에만. 전역 `~/.claude/`엔 진짜 범용만. 성능/규칙 개선 요청이 오면 **새 규칙·훅·메모리 추가가 기본이 아니다** — 먼저 무관 커넥터·플러그인·타 프로젝트 전역설정을 점검해 **스코프 교정·삭제로 수렴**한다(덧칠이 매번 다른 답·반복 실패의 원인 — 2026-07 확인).

## 2. 일하는 원칙
- **추측 금지.** 화면·에셋·좌표·파일 상태를 짐작 말고 실제 파일을 연다. 사용자가 언급한 파일은 답·수정 전 **먼저 읽는다.**
- 요청 범위 안 **가장 단순한 방법.** 요청 안 한 기능·리팩터링·스타일 변경 금지. **고장 안 난 코드·사용자가 만든 변경은 안 건드린다.**
- 실패한 테스트·타입에러·경고를 지우거나 임시로 숨기지 않는다. 비밀정보·토큰 노출 금지.
- **자주 하던 실수(꼭 점검):**
  - 비슷한 화면/방/이미지가 여럿일 때 겉보기(HUD 글자 등)로 단정 말고 **코드가 실제 로드하는 파일**부터 확인.
  - 빌드만 하고 끝내지 않는다 — 사용자가 쓰는 실행본(exe/app.asar)에 반영됐는지까지 확인.
  - 데이터 배열(맵 충돌격자 등) 수정 후엔 추측 말고 **실제 실행해 눈으로 검증.**
  - **PC마다 절대경로가 다름**(멀티PC). 작업일지·memory의 절대경로는 그 PC 기준 — "없다" 단정 전 `myPokemon_AJ/AGENTS.md`의 PC별 경로 + `find`로 실제 위치 확인. **PC간 규칙은 memory 말고 리포 AGENTS.md에 박는다.**

## 2.5 효율·성능 (시니어처럼 — 도구를 놀리지 않는다)
- **넓게 읽을 땐 서브에이전트 위임**(감사·탐색·"어디 있나" → `Explore`/general에 시키고 **결론만** 받음). 메인 컨텍스트에 파일 덤프 안 쌓아 판단력 유지. 독립 작업 여럿은 **한 번에 병렬** 발동.
- **의존 없는 도구 호출은 한 메시지에 묶어 병렬 실행.**
- **컨텍스트 위생(성능의 핵심):** 방금 편집한 파일 확인차 재통독 금지(Edit 실패하면 에러남). 큰 파일은 필요 구간만 offset. **긴 컨텍스트가 규칙을 묻는다** → 작업 전환 시 `/clear`, 길어지면 `/compact`.
- **큰 다분기·설계는 `Plan`/plan mode로** 뼈대부터 잡고 착수(헛발질 방지). 단순 작업엔 안 씀.

## 3. 검증 · Git
- **필수 게이트(건너뛰지 않음):** 코드 변경 → **`/verify`**(실동작) → 커밋 전 **`/code-review`**(effort medium) → **커밋 시 tsc 훅** 자동 통과(타입에러면 커밋 차단 — 무력화 금지). "완료"는 이 게이트 지난 뒤에만 말한다.
- 검증도구(이미 있음 — 새로 만들지 말 것): `/verify`, `/code-review`(품질만 정리=`/simplify`), **webapp-testing** skill(playwright, localhost:5180).
- **사용자에게 "확인해달라/봐달라" 할 땐 접속주소(`http://localhost:5180` + 어디로 가면 그 화면인지)를 항상 동봉**하고, 서버 200 확인 후 안내(exe로 볼 거면 실행법·경로). 매번 어디 볼지 되묻게 하지 말 것.
- 검증 못 한 부분은 "검증 못 했다"고 명확히. 실패하면 같은 시도 반복 말고 원인부터 다시 본다. 작업 끝나면 짧게 보고: **바꾼 파일 / 내용 / 한 검증 / 못한 것·남은 위험.**
- **승인 필요**(실행 전): 커밋·push·amend, 파일/디렉터리 삭제, 대량 이동·이름변경, 큰 구조변경·기능삭제, 외부전송·배포, 패키지설치·lockfile 변경. `git reset --hard`·`git checkout --` 등 사용자 변경 유실 명령은 명시적 승인 없이 금지.
