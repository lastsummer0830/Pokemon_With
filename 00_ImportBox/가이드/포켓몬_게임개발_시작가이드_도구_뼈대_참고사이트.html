01
Big Picture

먼저 전체 그림 — 우리가 만들 게임

코드부터 들어가기 전에 "무엇을 만드는가"를 한 문장으로 못 박아두면, 도구를 고를 때도 뼈대를 잡을 때도 헷갈리지 않아요. 우리 프로젝트의 한 문장은 이거예요.

🎯이 게임을 한 문장으로

"하트골드처럼 맵을 돌아다니며 포켓몬을 모으고 배틀하되, 내 집을 꾸미면 그 환경이 포켓몬의 컨디션을 만들고 그게 배틀로 이어지는 게임."

즉 포켓몬의 원래 컨셉(모으고 · 기르고 · 싸운다)은 그대로 두고, 본편에서 비어 있던 "같이 산다"는 축을 진하게 채우는 거예요. 집 꾸미기가 단순 장식이 아니라 배틀에 영향을 주는 순간, 흔한 클론들 사이에서 확 튀게 됩니다.

게임을 부품으로 쪼개보면

아무리 큰 게임도 결국 작은 부품들의 조합이에요. 하트골드 같은 2D 탑다운 RPG는 보통 이런 부품으로 나뉩니다. 처음부터 다 만들 필요 없이, 위에서부터 하나씩 쌓아 올리면 됩니다.

🗺️

① 맵 이동

타일(격자) 위에서 캐릭터가 걸어다니고, 벽·물에 막히는 충돌 처리. 제일 먼저 만드는 토대.
💬

② NPC · 대화

말 걸면 대사가 나오고, 이야기(스토리 플래그)가 진행. 하트골드의 "진행" 느낌이 여기서 나옴.
⚔️

③ 턴제 배틀

제일 덩치 큰 부품. 맨 마지막에 얹어야 안 지침. 서로 때리고 HP 깎는 단순 버전부터.
🏠

④ 집 꾸미기

가구를 격자에 놓고 그 배치를 저장. 내 색이 들어가는 핵심 차별 기능.
👕

⑤ 꾸미기(레이어)

옷·지닌 도구를 캐릭터/포켓몬 그림 위에 겹쳐 그리기. 코드는 쉽고 그림이 병목.
💾

⑥ 저장

잡은 포켓몬, 집 배치, 진행 상태를 파일/DB에 저장하고 다시 불러오기.
🪜"한 방에 다 만들기" 금지 — 쌓아 올리기

초보자가 게임 만들다 지치는 1번 이유는 배틀 같은 큰 부품을 맨 처음 손대는 것이에요. 우리는 ① 맵 이동이 되는 가장 작은 버전(수직 슬라이스)부터 만들고, 재밌다 싶으면 ②③… 순으로 살을 붙입니다. 각 단계가 끝날 때마다 "돌아가는 게임"이 손에 남는 게 핵심이에요.

🧠복습 질문

이 게임을 한 문장으로 말해보세요. 그리고 ①~⑥ 중 가장 먼저 만들 것과 가장 마지막에 만들 것은 각각 무엇인가요?

02
Decision

길 정하기 — 왜 Phaser + TypeScript인가

게임은 "어떤 언어 + 어떤 도구"로 만들지부터 정해야 해요. 결론부터 말하면 이번 프로젝트는 TypeScript(언어) + Phaser 3(게임 도구 모음) + VS Code(편집기) 조합을 추천합니다. 왜 그런지 천천히 풀어볼게요.

먼저 용어부터 — Phaser? TypeScript?

🍳비유 — 요리에 빗대면

Phaser(페이저) = 잘 갖춰진 주방 도구 세트. 게임을 맨손으로 만들면 "캐릭터를 화면에 그려라", "키보드 누르면 움직여라", "두 물체가 부딪쳤나 계산해라"를 전부 직접 코드로 짜야 해요. Phaser는 이런 자주 쓰는 기능을 미리 다 만들어두고 "가져다 써"라고 주는 도구 모음입니다. 그래서 우리는 게임 내용에만 집중할 수 있어요. 특히 2D(평면) 게임에 딱 맞고, 웹 브라우저에서 돌아갑니다.

TypeScript(타입스크립트, TS) = 안전장치가 달린 언어. 웹의 기본 언어는 JavaScript인데, TS는 거기에 "이 칸엔 숫자만 들어가야 해" 같은 규칙을 미리 정해두는 안전장치를 더한 버전이에요. 실수로 글자를 넣으면 컴퓨터가 미리 "잘못됐어!"라고 알려줍니다. 지금 배우는 Java도 똑같이 타입(자료형)이 있는 언어라, TS는 생각보다 금방 익숙해질 거예요.

정리하면 TS(언어)로 코드를 쓰고, Phaser(도구)로 2D 게임을 만든다 — 이런 조합입니다.

왜 이 조합인가 — 우리 목표에 맞는 이유

도구 선택은 유행이 아니라 "내가 만들 게임"과 "내 상황"에 맞춰야 해요. 우리 상황은 ① 2D 탑다운, ② 집 꾸미기 같은 자기만의 기능, ③ 포트폴리오로 남에게 쉽게 보여주기, ④ Java를 공부 중 — 이 네 가지죠.

🌐

링크 하나로 플레이

웹 게임이라 빌드해서 itch.io나 GitHub Pages에 올리면 "이 링크 눌러봐" 한 줄로 누구나 바로 해볼 수 있어요. 포트폴리오로는 이게 최강.
📚

참고 자료가 많음

Phaser로 만든 포켓몬 스타일 튜토리얼이 워낙 많아서, 막힐 때 따라 보며 갈 수 있어요. 요즘 AI 도움까지 더하면 충분히 완주 가능.
☕

Java 감각과 비슷

TS는 타입·클래스 개념이 Java와 닮아서, 지금 공부 흐름과 충돌이 적어요. 나중에 백엔드를 Spring으로 붙이기도 자연스럽고요.

03
Setup

필요한 프로그램(도구) 설치하기

이제 책상 위에 무엇을 깔아야 하는지 봅시다. 도구는 "꼭 필요한 것"과 "있으면 좋은 것"으로 나눠서, 처음엔 꼭 필요한 것만 깔면 됩니다.

꼭 필요한 것 (이것부터 설치)

⚙️

Node.js

역할: TS·Phaser 코드를 개발·실행하는 "엔진실". 설치하면 npm(부품 설치 도구)이 같이 깔려요. 없으면? Phaser 프로젝트를 아예 못 돌립니다. → 1순위 설치
🧰

VS Code

역할: 코드를 쓰고 고치는 편집기. TS·Git과 궁합이 좋고 가벼워요. 이미 써봤으니 그대로 사용.
🌿

Git + GitHub

역할: 코드의 "세이브 포인트"와 백업. 포트폴리오에선 "이 과정을 직접 만들었다"는 증거가 됩니다. 계정도 미리 만들어 두기.

있으면 좋은 것 (나중에 필요할 때)

🎨

Aseprite / LibreSprite

역할: 도트(픽셀) 그림 그리기 — 캐릭터·가구·아이템. Aseprite는 유료지만 편하고, LibreSprite는 무료 대체품. 처음엔 무료 에셋으로 시작해도 OK.
🗺️

Tiled

역할: 타일맵(맵) 그리기 도구. 그린 맵을 파일로 내보내면 Phaser가 그대로 읽어요. 맵이 커지면 큰 도움.
☕

IntelliJ / STS / Oracle

역할: 9번 "백엔드 저장" 단계로 갈 때만. 이미 익숙한 도구라 추가 학습 부담이 없어요.
🧭설치 흐름 — 위에서부터 차례로
1단계Node.js 설치 (LTS 버전 권장) → 컴퓨터에 "엔진실" 장착
↓
2단계VS Code 설치 → 코드 편집기 준비
↓
3단계Git 설치 + GitHub 가입 → 세이브/백업 준비
↓
4단계VS Code 확장 설치 (ESLint, Prettier 정도) → 코드 자동 정리

설치가 끝났는지 확인하는 명령. VS Code의 터미널(상단 메뉴 Terminal → New Terminal)에 한 줄씩 입력해 보세요.

터미널 (설치 확인)BASH
# Node.js와 npm이 깔렸는지 확인 — 버전 숫자가 나오면 성공
node -v
npm -v

# Git이 깔렸는지 확인
git --version
  • node -v는 "노드 버전 보여줘"라는 뜻. 버전 숫자(예: v20.x)가 보이면 정상 설치된 것.
  • npm은 Node와 함께 깔리는 부품 설치 도구예요. 다음 단계에서 Phaser를 이걸로 가져옵니다.
  • 셋 중 하나라도 "command not found"가 뜨면 그 프로그램이 설치 안 됐거나, 설치 후 터미널을 새로 안 연 것. VS Code를 껐다 켜고 다시 확인.
🚫자주 하는 실수

① 설치 직후 터미널에서 명령이 안 먹힘 → 보통 터미널(또는 VS Code)을 새로 열면 해결돼요. ② Node를 너무 최신/실험 버전으로 깔기 → LTS(안정 버전)를 받으세요. ③ Git만 깔고 GitHub 가입을 안 함 → 둘은 다른 거예요. Git은 내 컴퓨터의 도구, GitHub는 코드를 올려두는 웹사이트.

04
Reuse

예전 스윙 프로젝트 = 우리의 설계도

"예전에 스윙으로 간단히 해봤다"고 했죠. 그 프로젝트(javaprj의 swing_version)는 버리는 게 아니라 설계도로 재활용합니다. 이미 게임의 핵심 부품을 클래스로 나눠놨거든요. 그 구조를 Phaser/TS로 그대로 옮기면 됩니다.

📐비유 — 집을 옮겨 짓기

예전 스윙 코드는 "이 집에는 방이 몇 개, 부엌은 어디"가 그려진 설계도예요. 우리는 그 설계도(어떤 클래스가 무슨 일을 하는지)를 그대로 들고, 재료만 Java→TS로 바꿔서 새 땅(Phaser)에 다시 짓는 겁니다. 고민을 한 번 했던 구조라 두 번째는 훨씬 빨라요.

예전 클래스 → 새 구조 매핑

스윙 버전에 이미 있던 핵심 클래스들이 새 프로젝트에서 어디에 해당하는지 짝을 지어봤어요.

💡팁 — 코드가 아니라 "역할 분리"를 가져오기

옮길 때 Java 코드를 한 줄씩 직역하지 마세요. 가져올 건 "맵 / 플레이어 / 포켓몬 / 배틀 / 저장을 따로 둔다"는 역할 분리 감각이에요. 이미 그걸 해봤다는 게 큰 자산입니다. 새 기능인 집 꾸미기도 똑같이 "데이터(배치) / 화면(씬) / 저장"으로 쪼개서 끼워 넣으면 됩니다.

05
Skeleton

기본 뼈대 — 프로젝트 폴더 구조

게임이 커져도 안 헤매려면, 처음부터 "무엇을 어디에 둘지"를 정해두는 게 중요해요. 파일이 100개가 돼도 "포켓몬 데이터는 data 폴더, 화면은 scenes 폴더"라는 규칙이 있으면 길을 잃지 않습니다. 아래가 권장 뼈대예요.

📁my-pokemon/프로젝트 루트← 게임 전체가 들어가는 폴더
📄index.htmlHTML← 게임이 열리는 첫 웹페이지
⚙️package.json설정← 어떤 부품(Phaser 등)을 쓰는지 적힌 명세서
⚙️tsconfig.json설정← TypeScript 규칙 설정
📁src/소스← 우리가 직접 짜는 코드 전부

🔁이 구조에서 정보가 흐르는 길

게임이 켜지고, 집 꾸미기가 배틀에 영향을 주기까지의 흐름이에요. 폴더가 왜 이렇게 나뉘는지 이 흐름으로 보면 이해됩니다.

1main.ts가 게임을 켜고 WorldScene을 띄움
↓

↓

⚠️핵심 원칙 — "화면(scenes)"과 "정보(data)"를 섞지 않기

초보자가 가장 많이 하는 실수가 그리는 코드와 데이터·규칙을 한 파일에 다 욱여넣는 것이에요. 그러면 나중에 "배틀 공식만 고치고 싶은데" 할 때 화면 코드 사이에서 헤매게 됩니다. scenes(보여주기) · data(정보) · systems(규칙)을 나눠두면, 고칠 게 생겼을 때 어디를 열지 바로 알 수 있어요. 이게 예전 스윙 프로젝트가 이미 잘 해둔 부분(역할 분리)이고요.

06
Scene

게임의 뼈대 — Phaser 씬(Scene) 구조

Phaser 게임의 기본 단위는 씬(Scene)이에요. 씬은 게임 속 "한 장면"입니다. 우리 게임엔 맵 씬, 배틀 씬, 집 씬이 있고, 상황에 따라 씬을 갈아끼웁니다.

🎬비유 — 연극 무대

씬은 연극의 한 무대 세트예요. "마을 장면" 세트에서 놀다가, 배틀이 시작되면 무대를 "배틀 세트"로 바꿔 끼웁니다. 각 무대(씬)는 ① 소품을 준비하고(preload) → ② 무대를 차리고(create) → ③ 막이 오른 동안 계속 움직이는(update) 흐름을 가져요. 이 세 단계가 모든 씬의 공통 뼈대입니다.

모든 씬이 가지는 3단계

preload그림·소리 같은 리소스를 미리 불러오기 (딱 한 번)
↓
create화면을 처음 구성 — 캐릭터·타일·입력키 만들기 (딱 한 번)
↓
update1초에 약 60번 반복 — 매 순간 위치·상태 갱신 (게임 루프)

맵을 돌아다니는 가장 기본 씬. 예전 스윙의 MapPanel(그리기) + MapMain(키 입력)이 이 한 파일로 합쳐진다고 보면 돼요.

src/scenes/WorldScene.tsTypeScript
import Phaser from "phaser";  // Phaser 도구 모음을 통째로 가져온다

// Phaser.Scene을 상속받아 "우리만의 맵 장면"을 만든다 (Java의 extends와 같은 의미)
export default class WorldScene extends Phaser.Scene {
  private player!: Phaser.GameObjects.Sprite;            // 맵 위 내 캐릭터 그림
  private cursors!: Phaser.Types.Input.Keyboard.CursorKeys; // 방향키 입력

  constructor() {
    super("WorldScene");   // 이 씬의 고유 이름. 나중에 이 이름으로 씬을 부른다
  }

  preload(): void {        // 1) 리소스 먼저 불러오기 (딱 한 번)
    this.load.image("tiles", "assets/tilesets/town.png");
    this.load.spritesheet("hero", "assets/sprites/hero.png",
      { frameWidth: 16, frameHeight: 16 });
  }

  create(): void {         // 2) 화면 차리기 (딱 한 번)
    this.player = this.add.sprite(100, 100, "hero");
    this.cursors = this.input.keyboard!.createCursorKeys();
  }

  update(): void {         // 3) 매 순간 반복 (게임 루프 = 1초에 약 60번)
    if (this.cursors.left.isDown)  this.player.x -= 2;  // 왼쪽 키를 누르고 있으면 x를 줄임
    if (this.cursors.right.isDown) this.player.x += 2;
    if (this.cursors.up.isDown)    this.player.y -= 2;
    if (this.cursors.down.isDown)  this.player.y += 2;
  }
}
  • extends Phaser.Scene — Java에서 extends로 부모 클래스를 물려받듯, Phaser가 만든 "씬의 기본 기능"을 물려받아요. 그래서 우리는 preload/create/update만 채우면 됩니다.
  • super("WorldScene") — 이 씬에 이름표를 붙이는 부분. 배틀로 넘어갈 때 this.scene.start("BattleScene")처럼 이 이름으로 부릅니다.
  • update()가 1초에 약 60번 자동 반복된다는 게 게임의 핵심이에요. 예전 스윙에서 키 누를 때마다 한 번 다시 그리던 것과 달리, 여기선 "계속 도는 루프" 안에서 매 순간 상태를 확인합니다.
  • 다음 단계: player.x -= 2를 타일 격자 단위 이동 + 벽 충돌 막기로 바꾸면 하트골드 같은 칸 이동이 됩니다.
🚫자주 하는 실수

① 무거운 그림 로딩을 create나 update에 넣기 → 반드시 preload에서. ② update 안에 한 번만 해도 되는 일(버튼 만들기 등)을 넣기 → 매 프레임 60번 만들어져 버려요. ③ 씬 이름 오타 → this.scene.start로 호출할 때 이름이 한 글자라도 다르면 "그런 씬 없다"고 조용히 실패합니다.

07
Data

포켓몬 데이터 설계 — "정보 카드"

컴퓨터 입장에서 포켓몬 한 마리는 "정보가 적힌 카드" 한 장이에요. 이름·HP·기술이 적혀 있죠. 여기에 지닌 도구 칸과 컨디션 칸을 추가하는 게 우리 게임의 두 비밀 무기입니다.

🃏비유 — 트레이딩 카드

포켓몬 카드 한 장에 이름·타입·HP·기술이 적혀 있다고 생각해 보세요. 우리는 거기에 "지닌 도구: ___" 칸과 "컨디션: ___" 칸을 손으로 더 그려 넣는 거예요. 배틀할 때 그 칸을 읽어서 "이 칸에 기력의 조각이 있네? 그럼 공격력 +10" 식으로 효과를 줍니다. 원래 포켓몬의 "지닌 물건" 시스템과 똑같은 원리라 자연스럽게 녹아들어요.

포켓몬 한 마리의 정보를 정의하는 타입. interface는 "이 카드엔 이런 칸들이 반드시 있다"는 약속이에요.

src/data/Pokemon.tsTypeScript
// 포켓몬 한 마리가 가져야 하는 정보들의 "약속"
export interface Pokemon {
  id: number;            // 도감 번호
  name: string;          // 이름 (예: "파이리")
  type: string;          // 속성 (예: "불꽃")
  level: number;         // 레벨
  maxHp: number;         // 최대 체력
  currentHp: number;     // 현재 체력
  attack: number;        // 공격력
  heldItem: string | null; // ★ 지닌 도구 (없으면 null) — 비밀 기능의 핵심
  condition: number;     // ★ 컨디션(집에서 쉰 정도) — 집 꾸미기와 배틀을 잇는 값
}

// 새 포켓몬을 만들어 주는 함수 (스윙의 copy()/생성자 역할)
export function createPokemon(name: string, type: string): Pokemon {
  return {
    id: 0, name, type, level: 5,
    maxHp: 30, currentHp: 30, attack: 10,
    heldItem: null,   // 처음엔 아무 도구도 안 지님
    condition: 0     // 처음 컨디션은 0
  };
}
  • interface는 데이터의 "형태"를 정하는 약속이에요. 이렇게 정해두면 오타나 빠진 칸을 컴퓨터가 미리 잡아줘요 — TS의 안전장치(Java의 타입 검사와 같은 역할).
  • heldItem: string | null — "글자가 들어가거나, 아무것도 없으면 null"이라는 뜻. 도구를 안 지닌 상태를 깔끔하게 표현합니다.
  • condition이 8번 "집 꾸미기"와 4번 "배틀"을 잇는 다리예요. 집에서 잘 쉬면 이 숫자가 오르고, 배틀 계산이 이 숫자를 참고합니다.
🤫"지닌 도구를 포켓몬 위에 그리기"는 어떻게?

귀엽게 도구가 보이게 하는 건 옷 꾸미기와 똑같은 원리예요. 포켓몬 그림(아래 레이어) 위에 도구 그림(위 레이어)을 살짝 겹쳐서 그리면 됩니다. 데이터상으론 heldItem 칸에 글자만 들어 있으면 되고, 화면에선 그 글자에 맞는 작은 그림을 포켓몬 위에 얹는 거죠. 코드는 어렵지 않고, 도구 그림(에셋)을 준비하는 게 더 큰 일이에요.

🧠복습 질문

heldItem의 타입이 string | null인 이유는 무엇인가요? 그리고 condition 칸은 어떤 두 시스템을 연결하나요?

08
My Color ★

집 꾸미기 시스템 설계 — 내 색의 핵심

이제 이 프로젝트의 진짜 차별점이에요. 기술적으로 집 꾸미기는 어렵지 않아요. "격자 위에 가구를 놓고, 그 배치를 저장한다" — 동물의 숲 방 꾸미기와 똑같은 구조거든요. 핵심은 세 가지: 가구 목록 · 격자 배치 · 저장.

🪑비유 — 모눈종이 위에 스티커 붙이기

방을 모눈종이(격자)라고 생각하세요. 가구는 스티커고요. "이 스티커를 (3,2)칸에 붙였다"는 기록만 모아두면, 나중에 그 기록대로 다시 붙여서 방을 통째로 복원할 수 있어요. 즉 우리가 저장할 건 그림이 아니라 "무슨 가구를 몇 번 칸에 놨는지 목록"뿐입니다.

집 배치를 표현하는 데이터. 이 작은 목록 하나만 저장하면 방 전체가 복원돼요.

src/data/HouseLayout.tsTypeScript
// 가구 하나가 "어디에 놓였는지" 한 칸 기록
export interface PlacedFurniture {
  itemId: string;  // 가구 종류 (예: "fireplace" = 벽난로)
  x: number;       // 방 격자 칸 x 좌표
  y: number;       // 방 격자 칸 y 좌표
}

// 방 한 칸 전체 = 크기 + 놓인 가구 목록
export interface HouseLayout {
  width: number;                 // 방 가로 칸 수
  height: number;                // 방 세로 칸 수
  furniture: PlacedFurniture[];  // 놓아둔 가구 목록 (이 배열만 저장하면 방이 통째로 복원됨)
}
  • PlacedFurniture[]의 []는 "여러 개를 담는 목록"이라는 뜻 (Java의 List와 같은 역할).
  • 가구 그림이 아니라 "종류 + 좌표"만 저장한다는 게 핵심. 그래서 저장 용량도 작고, 불러올 때 그 좌표에 그림을 다시 그리기만 하면 됩니다.
  • 놓을 수 있는 가구의 그림·이름·효과는 furniture.ts(카탈로그)에 따로 정의해두고, 여기선 itemId로 가리키기만 해요.

★ 진짜 차별점 — 집이 배틀에 영향을 준다

여기가 "와 이거 신박하다"가 나오는 지점이에요. 집 꾸미기를 단순 장식이 아니라, 포켓몬 컨디션을 올리고 그게 배틀로 이어지게 만듭니다. 어제 얘기했던 "불꽃 포켓몬을 벽난로 옆에서 재우면 강해진다"를 코드로 옮기면 이래요.

집 환경을 보고 포켓몬 컨디션을 올려주는 규칙. systems/ 폴더에 둬서 배틀·화면과 분리.

src/systems/homeBonus.tsTypeScript
import { Pokemon } from "../data/Pokemon";
import { HouseLayout } from "../data/HouseLayout";

// 집에서 쉴 때 호출 — 방 안의 가구를 보고 컨디션을 조정한다 (내 색의 핵심 규칙)
export function applyHomeBonus(pokemon: Pokemon, house: HouseLayout): void {
  // 방에 벽난로가 하나라도 있는지 확인
  const hasFireplace = house.furniture.some(f => f.itemId === "fireplace");

  if (pokemon.type === "불꽃" && hasFireplace) {
    pokemon.condition += 10;  // 불꽃 포켓몬 + 벽난로 → 컨디션 +10
  }

  // 물 포켓몬은 연못이 있으면 컨디션 상승 (같은 방식으로 규칙을 늘려가면 됨)
  const hasPond = house.furniture.some(f => f.itemId === "pond");
  if (pokemon.type === "물" && hasPond) {
    pokemon.condition += 10;
  }
}
  • house.furniture.some(...) — 목록 중 "조건에 맞는 게 하나라도 있나?"를 묻는 것. 벽난로가 있으면 true.
  • 타입(불꽃/물)과 가구(벽난로/연못)를 짝지어 컨디션을 올려요. 이 if를 늘려가는 것만으로 "집이 의미를 갖는" 규칙이 쌓입니다.
  • 이렇게 올린 condition을 systems/battle.ts의 데미지 계산에서 참고하면, "잘 꾸민 집 → 강한 배틀" 고리가 완성돼요.
🔗두 시스템이 이어지는 한 줄 흐름
꾸미기불꽃 포켓몬 방에 벽난로를 놓는다 (HouseScene)
↓
휴식집에서 쉬면 applyHomeBonus가 condition +10
↓
배틀battle.ts가 condition을 반영 → 살짝 더 강하게 싸움
↓
의미"내가 꾸민 집이 내 포켓몬을 강하게 만든다" — 사람들이 진심으로 꾸미기 시작
🌱작게 시작하기 — 한 번에 다 만들지 말 것

집 꾸미기도 "가구 1종을 1칸에 놓고 화면에 보이게"부터 시작하세요. 그게 되면 → 가구 여러 개 → 저장/불러오기 → 컨디션 규칙 → 배틀 반영 순으로. 어제 정한 "할만하면 하나씩 추가" 전략 그대로예요.

09
Optional · Full-stack

(선택) 스프링부트 백엔드로 저장하기

이건 나중에 여유가 생기면 붙이는 선택 단계예요. 게임 화면은 Phaser로 두되, 잡은 포켓몬·집 배치 같은 데이터를 Spring Boot + MyBatis 백엔드(서버)에 저장하면, 지금 공부 중인 스프링이 "학습용"이 아니라 "실제 프로젝트에 쓰였다"는 강력한 포트폴리오 서사가 됩니다.

🏦비유 — 게임은 매장, 서버는 금고

지금까지는 저장을 브라우저(내 컴퓨터)에 했어요. 매장 금고에 보관하는 셈이라, 다른 기기에선 못 봐요. 백엔드를 붙이면 은행 금고(서버 DB)에 보관하는 거라, 어디서 접속해도 내 집·포켓몬이 그대로 있습니다. 게임(매장)이 금고(서버)에 "이거 저장해줘 / 꺼내줘"라고 부탁하는 구조죠. 이 흐름이 바로 어제 정리한 MyBatis 자료의 "사서에게 부탁하는 흐름"과 같아요.

게임(Phaser)집 배치를 JSON으로 서버에 보냄 (fetch)
↓

서버가 집 배치를 받을 때 쓰는 데이터 그릇(DTO). 게임이 보낸 JSON이 이 모양으로 담겨요.

HouseLayoutDto.javaJava
package com.mygame.house;   // 이 클래스가 속한 패키지(폴더) 이름

import java.util.List;        // 목록(List)을 쓰기 위해 가져옴

// 게임에서 보낸 "집 배치"를 담는 그릇 (TS의 HouseLayout과 같은 정보)
public class HouseLayoutDto {
    private long playerId;                 // 어느 플레이어의 집인지
    private List<PlacedFurnitureDto> furniture; // 놓인 가구 목록

    // getter/setter — 외부에서 값을 넣고 꺼내는 통로 (스프링이 JSON을 자동으로 채워줌)
    public long getPlayerId() { return playerId; }
    public void setPlayerId(long playerId) { this.playerId = playerId; }
    public List<PlacedFurnitureDto> getFurniture() { return furniture; }
    public void setFurniture(List<PlacedFurnitureDto> furniture) { this.furniture = furniture; }
}
  • 게임(TS)의 HouseLayout과 서버(Java)의 HouseLayoutDto는 같은 정보를 양쪽 언어로 적은 것이에요. 모양이 맞아야 JSON이 깔끔하게 오갑니다.
  • 스프링은 게임이 보낸 JSON을 보고 이 DTO의 칸을 자동으로 채워줘요. 그래서 우리는 그릇만 만들어두면 됩니다.
  • 이 DTO를 Repository → Mapper XML로 넘기면, 어제 정리한 MyBatis 흐름 그대로 DB에 저장됩니다.
⏳순서 주의 — 이건 마지막에

백엔드는 게임이 어느 정도 돌아간 뒤에 붙이세요. 처음부터 서버까지 같이 만들면 둘 다 미완성으로 지칩니다. "저장이 필요하네" 하는 순간이 올 때 이 단계로 오면 돼요. 그전까지는 8번까지의 브라우저 저장으로 충분합니다.

배포 · 포트폴리오

  • itch.io https://itch.io/ — 웹 게임을 올려 링크로 공유.
  • GitHub https://github.com/ — 소스 코드 + README로 "내가 만든 증거".
⚖️반드시 — 에셋 라이선스와 "포켓몬" 저작권

① 무료 에셋도 "표기 필요 / 상업 이용 가능" 같은 조건이 제각각이에요. 쓴 에셋은 출처·라이선스를 README에 기록해 두세요(어제 정리 md의 "에셋 라이선스 기록 예시" 참고). ② "포켓몬"은 닌텐도의 상표·저작물이라, 공개 배포·포트폴리오용으로는 실제 포켓몬 이름·이미지·음악을 그대로 쓰면 위험합니다. 학습/개인 연습은 괜찮지만, 남에게 보여줄 땐 "포켓몬에서 영감받은 나만의 몬스터·세계관"으로 이름과 그림을 바꾸는 걸 권해요. 그게 오히려 "내 색"을 더 분명히 보여주기도 하고요.
